## 📋 项目全流程说明

### **一、核心业务流程**

#### **1. 内容导入流程**
```
用户上传文档 → 文档解析 → 元数据提取 → 分块处理 → 向量化 → 存储到Milvus → 任务状态追踪
```

- **支持内容类型**：景区攻略、线路推荐、酒店信息、美食推荐、交通指南
- **元数据管理**：景区名、线路名、地区名、来源文件名、内容类型等
- **质量保障**：任务状态追踪，避免导入失败导致服务不可用

---

### **二、用户查询流程（4大场景）**

#### **场景1：景点介绍**
```
用户提问（如"三亚有哪些必去景点"）
    ↓
意图识别 → 景点介绍类
    ↓
检索知识库 → 按目的地/景点名称筛选
    ↓
返回结果：景点列表 + 景区特色 + 适合人群 + 最佳游览时间
```


#### **场景2：景点详情查询**
```
用户指定景点（如"天涯海角门票多少钱"）
    ↓
意图识别 → 实用信息查询
    ↓
精确定位 → 景点文档片段
    ↓
返回结果：简介 + 开放时间 + 门票价格 + 游览路线 + 注意事项 + 实用Tips
```


#### **场景3：线路检索**
```
用户询问行程（如"三亚3天怎么玩"）
    ↓
意图识别 → 线路检索类
    ↓
定位线路文档 → 提取行程片段
    ↓
返回结果：线路名 + 行程天数 + 预算参考 + 来源信息（可结构化展示）
```


#### **场景4：实用信息查询**
```
用户查询（如"带老人孩子适合去哪玩"）
    ↓
意图识别 → 知识问答/实用信息
    ↓
混合检索 → 景点+线路+酒店+美食+交通
    ↓
返回结果：详细信息 + 推荐建议 + Tips + 注意事项 + 来源追溯
```

### **三、技术架构流程**

```
┌─────────────┐
│  用户输入    │
└──────┬──────┘
       ↓
┌─────────────┐
│  意图识别    │ → 4类：景点介绍/线路检索/实用信息/知识问答
└──────┬──────┘
       ↓
┌─────────────┐
│  文档解析    │ → Markdown按二级标题分块 + 元数据提取
└──────┬──────┘
       ↓
┌─────────────┐
│  向量检索    │ → Embedding → Milvus检索相关片段
└──────┬──────┘
       ↓
┌─────────────┐
│  回答生成    │ → 基于检索内容生成答案 + 引用溯源
└──────┬──────┘
       ↓
┌─────────────┐
│  流式输出    │ → 实时返回 + 历史记录管理
└─────────────┘
```


---

### **四、数据流转全链路**

```
原始Markdown文档
    ↓
【分块处理】按二级标题切分
    ↓
【元数据标注】内容类型/城市/景点名/来源等
    ↓
【向量化】Embedding模型生成向量
    ↓
【存储】Milvus向量数据库
    ↓
【检索】语义搜索匹配用户query
    ↓
【排序】相关性评分 + 元数据过滤
    ↓
【生成】LLM整合信息生成回答
    ↓
【输出】流式返回 + 引用标注
```


---

### **五、关键质量要求**

✅ **异常处理**：导入失败不影响整体服务  
✅ **准确性**：查询结果可解释、可追溯  
✅ **实时性**：流式输出具备良好响应速度  
✅ **可扩展**：模块清晰分层，便于后续扩展  

---

### **六、使用角色对应流程**

| 角色     | 典型操作                   | 系统响应                    |
| -------- | -------------------------- | --------------------------- |
| **游客** | 查景点/线路/酒店/美食/交通 | 返回详细介绍+推荐+实用信息  |
| **导游** | 快速检索景区资料           | 精准定位文档片段+结构化答案 |
| **运营** | 上传文档/校验内容          | 任务状态追踪+元数据管理     |

这个流程设计体现了**RAG（检索增强生成）**的核心思想，通过意图识别→文档解析→向量检索→答案生成的流水线，实现准确、可追溯的智能旅游问答服务。
